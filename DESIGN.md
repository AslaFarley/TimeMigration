# 《时间移民》(TimeMigration) — 游戏设计与技术规格文档

> 本文档是项目的权威设计规格。任何模型/开发者接手时，请先完整阅读本文件。
> 灵感来源：刘慈欣《时间移民》。核心玩法：携带冷冻人口在未来不同年代间"跳跃移民"，
> 通过对未来下注、用睡眠时长赌一个好时代，最终为人类文明找到栖身之所。

---

## 0. 技术选型（已定稿）

- **前端**：React + TypeScript + Vite（纯静态、组件化、桌面浏览器）
- **后端**：第一版**不做**。代码中预留 `NarrativeProvider` / `EndingProvider` 接口，
  未来可接后端代理 + 实时 LLM（方案：D 离线打底 + B 后端代理可选升级）。
- **内容**：手写种子库 + 附带一个可选的 LLM 批量生成脚本。
- **范围（MVP）**：完整可玩的核心循环；不做存档 / 排行榜 / 音效。
- **平台**：桌面浏览器（鼠标键盘）。

---

## 1. 核心玩法循环

```
开局：携带 frozenPop（冷冻人口）抵达"时间彼岸"，进入第一个时代
  ↓
[决策阶段 · 限时 60 秒]
  阅读当前时代的"叙述文本"（含可控噪声/矛盾）+ 历史记录 + 先遣队情报
  做出最多 3 个决策：
    Q1. 是否全体解冻、定居此时代？
        ├─ 是 → 进入结局结算（把剩余 frozenPop 全部投入当前时代）
        └─ 否：
            Q2. 选择本次睡眠时长（影响时代跨度与风险）
            Q3. 是否派先遣队
  ↓
[沉睡过渡] 播放氛围文字（预留为 LLM 加载缓冲）
  ↓
[苏醒结算] 世界按状态机演化，揭晓真实情况 vs 当时叙述
  ↓
回到决策阶段，直到玩家选择"全体解冻"或触发失败结局
```

---

## 2. 玩家输入模型（关键，仅 3 个决策）

玩家**不直接控制 activePop（已解冻人口）**。每轮最多 3 个输入：

1. **是否全体解冻定居**（是 → 结束游戏并结算结局）
2. **本次睡眠时长**（否定居时才需要）
3. **是否派先遣队**（否定居时才需要）

> activePop 的增减完全由"先遣队归队 + 世界参数演化"驱动，玩家不直接设定。

---

## 3. 先遣队（scoutForce）机制

- **固定数量**：每次派遣解冻**固定数量**的人，从 `frozenPop` 中扣减。
- **不选任务**：玩家只决定"是否派"。任务由系统**自动选择最优**：
  系统判断下一个时代哪个参数（acceptance / habitability）对人口存活最关键，
  就让先遣队去改造那个参数。
- **双重价值**：
  1. **注入人口**：先遣队在目标时代存活后并入 activePop。
  2. **改造未来**：自动最优任务小幅提升下一个时代的对应参数。
  3. **带回情报**：先遣队的存活率（归队数 / 派出数）是**比叙述文本更可靠的真实损耗率情报**，用于对抗叙述噪声。

---

## 4. 人口演化公式（每次苏醒结算）

```text
设 f = f(habitability, acceptance, tech)   // 人口乘子：环境好 >1（壮大），差 <1（损耗）

// 1) 先遣队（上次派出的）在本时代存活并归队，注入主体
scoutReturn = scoutSent * f
activePop  += scoutReturn

// 2) 已在此世界的人口随当前时代环境演化
activePop  *= f

// 3) 先遣队任务效果（自动最优）作用于"下一个时代"的对应参数
nextEra[targetParam] += scoutMissionBonus
```

- 玩家**唯一主动增加 activePop 的手段 = 派先遣队**。
- 不派先遣队 → activePop 仅随世界参数自然增减。
- 「全体解冻」= 将剩余 `frozenPop` 全部并入 `activePop` 并在当前时代结算结局。

---

## 5. 世界状态机参数（5 个核心参数）

| 参数 | 字段 | 含义 | 作用 |
|---|---|---|---|
| 宜居度 | `habitability` (0-100) | 客观环境好坏（**天灾维度**：气候/辐射/资源/生态） | 影响人口乘子 f |
| 接纳度 | `acceptance` (0-100) | 原住文明对"过去来的移民"的欢迎度（**人祸维度**） | 影响 f 与 trust |
| 科技 | `tech` (0-100) | 时代科技水平 | 影响 f 与 capacity |
| 承载力 | `capacity` (数值) | 该时代能容纳的人口上限 | 超载触发**渐进**冲突损耗 |
| 民心 | `trust` (0-100) | 替代"瞬间政变死亡" | 低于阈值有预警，归零才失败 |

> 宜居度 vs 接纳度：**宜居度低 = 天灾；接纳度低 = 人祸**。两者独立，组合出不同时代质感。

**玩家资源**：
- `frozenPop`：剩余冷冻人口（先遣队来源池）。
- `activePop`：已解冻、在当前时代定居的人口。
- `scoutForce`：当前派出在外、尚未归队的先遣队人数。

---

## 6. 失败 / 结局（渐进预警，非瞬间死亡）

- **民心崩溃**：`trust` 低于阈值 → 先出现预警（数轮缓冲）→ 归零才失败（政变）。
- **人口超载**：`activePop > capacity` → 先警告资源紧张 → 持续超载才触发冲突损耗。
- **主动结局**：玩家选择「全体解冻」→ 触发结局结算。

### 结局系统（开放式，非确定性评分）
- **不给数字评分**，避免确定性结局感。
- 第一版：显示**成功 / 失败 + 一段简短的规则化定性描述**。
- 未来（预留）：用 LLM 根据"最终世界状态 + 整局历史轨迹 + 玩家所有决策"
  生成一段**"移民文明苏醒后 1000 年发展历程"的叙事**，成败由玩家自行解读。
- **从第一版起完整记录历史轨迹**，确保未来切 LLM 时上下文数据现成。

---

## 7. 叙述系统（核心设计：把 LLM 不稳定变成 feature）

- **NarrativeProvider 接口**：`getNarrative(worldState, era) => NarrativeText[]`
- **第一版实现 `SeedLibraryProvider`**：从手写种子库按时代类型选片段。
- **受控噪声注入（`noise.ts`）**：代码层面持有"真值"，根据当前时代的
  "叙述者类型"对外显示文本注入可控偏差。叙述者类型示例：
  - 乐观夸大型（OPTIMIST）：把数值往好里说
  - 悲观型（PESSIMIST）：把数值往坏里说
  - 撒谎型（LIAR）：故意给出与真值矛盾的信息
  - 可靠型（RELIABLE）：基本如实
  玩家通过多轮苏醒对比，**学会识别叙述者风格**——这让 LLM/文本的不稳定变成玩法。
- **历史记录（HistoryPanel）**：每次苏醒记录「当时叙述 vs 实际真值」，可回看。
- **先遣队情报**：派过先遣队的时代，给出比公开文本更准的"内部报告"（真实损耗率）。
- **预留 `LLMProvider`**（走后端代理）作为未来替换项，核心玩法不依赖它。

---

## 8. 时代类型库（满足"远超 10 种年代"）

状态机决定抽到哪类，种子库为每类提供多条描绘：

黄金时代 / 战乱 / 生态崩溃 / 科技奇点 / 伪繁荣 / 寒冬 / 瘟疫 / 复兴 / 极权 / 真空荒芜 …（可持续扩充）

每种时代类型定义其参数生成倾向（如"伪繁荣"= 叙述乐观但真实 trust/habitability 偏低）。

---

## 9. 沉睡过渡界面（SleepTransition）

- 在「决策确认 → 苏醒结算」之间插入过渡界面，播放氛围文字（淡入淡出），例如：
  - "先遣队已出发，载着希望驶入未知的时间长河……"
  - "你重新进入深度睡眠，思绪万千，不知再睁眼时世界是何模样……"
  - "冷冻舱的低鸣渐渐远去……"
- 实现：组件接收一个 `Promise`（内容就绪信号）。
  - 第一版传一个定时 Promise（2-3 秒纯氛围）。
  - 未来接 LLM：换成 LLM 请求 Promise，把过渡时间用作真实加载缓冲，UI 不变。

## 11. 实施步骤

1. 初始化 Vite + React + TS 项目，安装依赖
2. 定义 `types.ts` 数据模型 + 时代类型库 `eras.ts`
3. 实现 `worldEngine`（人口公式 / 演化 / 先遣队自动最优任务 / 渐进失败）
4. 实现叙述系统（接口 + 种子库 + 噪声注入 + 手写种子）
5. 实现结局系统 `ending/`（接口 + 规则实现，预留 LLM）
6. 实现游戏状态管理 `gameStore`（含完整历史轨迹记录）
7. 实现 UI 各屏幕（3 步决策含 60s 计时 → 沉睡过渡 → 苏醒 → 开放式结局）
8. 串联完整一局循环，跑通可玩
9. 编写可选的批量生成脚本
10. 本地启动验证
